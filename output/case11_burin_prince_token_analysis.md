# 케이스11 실행 분석 — 「부린 왕자」 (검색기: token / V1)

> gold: 작성자확정 **179.9** → 교열최종 **332.6324** (뒤집힘) / 난이도 **상** (문서 전체 최상위)
> 결과: **650.1 1순위 (0.45, 저확신) · Exact ✗ · Top-k ✗** · **Escalate ✓** (needs_toc + cross_major, ambiguity 0.80)

## 1단계 (retrieve, 제목 검색) 평가 — **실패** ❌ (V1의 한계가 적나라하게 드러남)

- 제목 "부린 왕자"는 '부동산+어린왕자'의 **조어(coined)**. 토큰 {부린, 왕자}가 DB 어느 제목과도 2개 이상 안 겹침 → **제목 매칭 0건.**
- 결과: 후보가 **082(650.1) 하나뿐.** 사서가 실제로 검토한 179.9·332.6324·811.36은 **아예 후보에 안 올라옴.**
- 즉 정답(332.6324)이 후보에 없으니, 2단계가 아무리 잘해도 **구조적으로 못 맞힘.** 이건 classify 문제가 아니라 **retrieve recall 실패.**
- → 우리가 예측한 "제목 글자가 안 겹치면 못 잡는다"가 가장 극단적으로 실현. **V2(임베딩)** 또는 **입력 보강(부제/키워드/책소개)** 없이는 이 케이스 접근 불가.

## 2단계 (classify, LLM 판단) 평가 — **빈약한 입력에도 안전하게 행동** ✅

후보가 650.1 하나뿐인 열악한 상황에서 LLM은:
- 650.1을 1순위로 두되 **confidence 0.45로 낮춤** (확신 안 함).
- **정확히 에스컬레이션**: `needs_toc`(부제·목차·키워드 전무로 초점 판별 불가) + `cross_major`(소설 800대 / 부동산 330대로 갈릴 수 있음). ambiguity 0.80.
- 종합 판단 근거: "'부린 왕자'는 어린왕자 패러디로 보이나 메타데이터가 없어 초점 판별 불가 → 목차/책소개 확보 후 사서 최종 판단 필요."
- → 대분류 교차 가능성을 스스로 감지하고 사람에게 넘김. **상 케이스에 맞는 동작.** (단, gold를 후보에 못 넣은 건 1단계 탓)

## 종합 — 이 케이스가 알려주는 것

- **병목은 1단계(retrieve recall).** 조어 제목 + 메타데이터 부재 → 후보 생성 실패. classify는 그 위에서 할 수 있는 최선(저확신 + 에스컬레이션)을 했다.
- **평가 관점:** 상 케이스는 "정답 하나 맞히기"(Exact)로 재면 필연적으로 실패(정답이 후보에 없거나, 사람도 갈림). 여기서 유의미한 성공 지표는 **"에스컬레이션을 제대로 띄웠나"** — 그건 ✓. 즉 시스템은 "모르는 걸 모른다고 하고 사람에게 넘기는" 안전망으로선 작동했다.
- **개선 방향:** (a) 입력에 부제/책소개/키워드를 넣으면 1단계가 후보를 건질 수 있는지, (b) V2 임베딩이 조어 제목의 의미('부동산','성찰')를 잡아 후보를 넓히는지 — 두 실험의 직접 대상.
