# 케이스5 실행 분석 — 「가장 인간적인 도시」 (검색기: token / V1)

> 실행: `python src/run.py` (RETRIEVER=token). 산출물: `case05_humane_city_token_{retrieve,result,run}.json`
> 정답(gold): 작성자확정 **720.2** / 교열최종 **720.2** / 난이도 **중**

## 결과 요약

| 항목 | 값 |
|---|---|
| 입력 082 | 720.105 |
| 1차(토큰) 후보 번호 | 720.105(★082, 매칭0) · 720.2(매칭3) · 711.4(매칭1) |
| 2차(LLM) 최상위 | **720.105** (적합도 0.62, 082_kept) |
| 2차 후보 전체 | 720.105 → 711.4 → 720.2 |
| 에스컬레이션 | ✅ `needs_toc` (HITL) |

**지표:** Exact ✗ (720.105≠720.2) · Prefix ✓ (앞 3자리 720 일치) · **Top-3 ✓** (720.2가 3순위) · Escalate 적절

## 무슨 일이 일어났나

LLM은 gold(720.2)가 아니라 **720.105를 1순위로, 082를 유지(kept)**했다. 왜?

- 우리가 크롤한 720.105 서가 샘플이 **「인공지능시대의 건축: AI 입문」, biomimetics/responsive architectures** 등 *AI·첨단기술 건축*을 담고 있다.
- 입력 도서 부제가 정확히 **"AI시대 건축"**이라, LLM은 "720.105 = AI건축 → 입력과 주제 밀착 → 082 유지"로 판단했다.
- 반면 720.2 샘플은 건축 기행·집과 인문학 에세이 등 **대중 교양** 성격이라, LLM은 "AI 초점과 다소 어긋남"으로 3순위로 내렸다.

## 왜 gold는 720.2인가 (사람 사서의 판단)

시나리오 문서: *"청구기호 검색 → 본교서지: 720.105는 **공공성 관련** 포함 10권, 도시·건축과 align 안 함 → 기각. 제목 검색 분포 720.2 최다 + 본교 의미 '도시+건축+공간' → 720.2 확정."*

즉 **사서는 720.105를 '공공성'으로 보고 기각**했는데, **우리 LLM은 720.105를 'AI건축'으로 보고 채택**했다. 정반대 판단.

## 핵심 통찰 ⭐ — LLM의 판단은 "retrieve가 보여준 샘플"에 좌우된다

같은 번호(720.105)를 두고 사서와 LLM이 반대로 간 이유는 **본 샘플이 달랐기 때문**이다.
- 사서가 근무 중 본 720.105 = "공공성" 성격 (그 시점 상위 노출).
- 우리 LLM이 본 720.105 = "AI건축" 성격 (우리 크롤 첫 8권이 AI/이론서).

→ **1차(retrieve)가 어떤 책을 shelf_sample로 뽑아 보여주느냐가 2차(LLM) 판단을 바꾼다.** 이건 버그가 아니라 **설계 지렛대**다:
- 지금은 `shelf_sample = 앞 8권`(임의). 이게 편향되면 LLM 판단도 편향된다.
- 개선 여지: 번호대를 **대표하는** 책을 뽑기(무작위/다양성 샘플), 또는 더 많이 보여주기.

이건 앞서 devlog에 적은 "실데이터가 케이스를 더 어렵게 만든다(720.105에 AI건축 책 실재)"가 **실제 판단 오차로 구현된** 사례다.

## 무엇이 설계대로 잘 작동했나

1. **082 prior 강제 포함** — 720.105는 제목매칭 0이지만 prior라 후보에 올랐고, LLM이 그걸 두고 유지/기각을 판단했다. (082 라우팅 의도대로)
2. **"개수 아니라 주제 우선"이 실제로 작동** — 720.2가 제목매칭 최다(3)였지만 LLM은 개수 대신 **주제**로 720.105를 골랐다. 지침대로 움직였다. (다만 이번엔 그 판단이 gold와 갈림 → 양날)
3. **에스컬레이션 정확** — `needs_toc`로 HITL 올림. 실제로 이 케이스는 사서·LLM 판단이 갈리는 애매 케이스라, "단정 말고 사람에게"가 맞다. Top-3 안에 gold 있음 → 사서가 720.2를 고를 수 있음.

## 무엇이 V1(토큰)의 한계로 드러났나

- **제목 매칭이 성김 (총 4건).** 한국어 조사가 토큰에 붙어("건축가가"≠"건축") 매칭이 약하다. 실제 후보였던 **720.1이 아예 안 떠올랐다**(제목 겹침 0).
- 즉 recall이 새고, 후보 폭이 좁다. → **V2(임베딩)**가 "AI시대"↔"인공지능시대", "도시"↔"도시디자인" 같은 의미 유사를 잡아 후보를 넓히는지 비교할 지점.

## 재실행 (shelf_sample 캡 8→40, 표시 [:6]→전체)

이전엔 classify가 `shelf_sample[:6]`으로 잘라 720.105의 7번째 「프라이버시와 공공성」을 LLM이 못 봤다.
캡을 풀어 **7권 다 보여주고 재실행**한 결과:

- **여전히 720.105 1순위(082 유지, 0.62).** → 잘린 책이 결정적 원인이 아니었음. 공공성 책까지 봐도 LLM은 "AI건축 6/7 + 부제 'AI시대 건축'"으로 720.105를 택함. (이전 분석의 "컷이 판단을 흔들었다"는 과한 해석이었음.)
- **그러나 개선은 실재:** 711.4 근거가 더 대표성 있게 바뀜(앞8권 "도시생태현황도" → 40권 보니 「보행도시」「살고싶은 마을 만들기」「도시 클리닉」 → "인간 중심 도시"로 해석, 실제 제목과 더 가까움). **자를 이유 없었음이 확인.**

### 진짜 원인 (재규명) — 샘플 편향이 아니라 '판단 축'의 차이
- LLM: **주제 키워드**로 매칭 → "AI건축" → 720.105(AI책 있음) 유지.
- 사서: **책의 성격/급**으로 판단 → "실리콘밸리 건축가가 안내하는" = 대중 안내서 → 이론서가(720.105) 아닌 일반 건축/도시(720.2).
- 720.105는 건축 이론·전문서 서가라, AI를 다뤄도 *대중 안내서*는 720.2로. LLM은 이 '급/장르' 축을 못 잡음.
- → 개선 지렛대: classify 프롬프트에 "주제뿐 아니라 **책의 성격(이론 전문서 vs 대중 교양·안내서)**도 고려" 신호 추가 검토.

## 3차 실행 — classify 프롬프트에 '책의 급/장르' 축 추가

가설: LLM이 '주제 키워드'만 보고 '책의 급(전문 이론서 vs 대중 안내서)'을 안 본다.
조치: classify 시스템 프롬프트에 "주제뿐 아니라 책의 성격/급도 보라. 이론 모노그래프 서가 vs 대중 교양·안내서 서가는 번호가 다르다"를 추가.

**결과: 720.2로 뒤집힘 = gold 일치.** Exact✓ / Prefix✓ / Top-3✓ / Escalate✓.
- [1] 720.2 (0.60) — "'안내하는' 대중 교양서라 교양 서가가 급이 맞다"
- [2] 720.105 (0.35, 082) — "AI 주제는 겹치나 이론서가라 급이 다름. 목차상 AI 기술 본격이면 이 번호"로 병기·강등
- [3] 711.4 (0.20)

→ **가설 검증 완료.** divergence의 원인은 shelf_sample 편향이 아니라 **'급/장르' 축의 부재**였고, 프롬프트 한 줄로 교정됨.

### ⚠️ 과적합 경계
케이스5 하나를 gold에 맞춘 것. "대중서 → 일반 번호대" 규칙이 **정답이 전문/세분 번호인 케이스를 반대로 망칠 위험**이 있다(예: 향신료 641.338309 같은 세분 번호). → 다른 케이스로 일반화 검증 필수. 골든셋을 늘려 이 프롬프트가 전체 정확도를 올리는지 확인해야 함.

## 1단계 / 2단계 평가 요약

- **1단계(retrieve):** 부분 성공. 720.2(정답)를 매칭 3건으로 건짐 + 082(720.105) prior 포함. 단 720.1은 토큰 안 겹쳐 누락. 한국어 조사로 매칭이 성김(총 4건). → recall은 정답을 놓치진 않았으나 좁음.
- **2단계(classify):** '급/장르' 축 추가 후 대체로 720.2를 고름(아래 비결정성 참조). 720.2를 항상 후보에 유지 + 항상 에스컬레이션 → 시스템 동작은 안정.

## ⚠️ 비결정성 발견 (중요)

'급/장르' 프롬프트 추가 후 케이스5를 여러 번 돌린 결과:
- run.py ×5 → **720.2**, evaluate ×1 → **720.105**. 즉 **~5/6 720.2(정답), ~1/6 720.105.**
- LLM(thinking adaptive) 확률성 때문에 **애매 케이스는 top pick이 실행마다 흔들린다.**
- 프롬프트는 "항상 틀림(720.105) → 대부분 맞음(720.2)"으로 개선했지만 **결정론적이진 않다.**
- **안정적인 신호:** 720.2가 **항상 Top-k 후보에 있음** + **항상 에스컬레이션**. → 애매 케이스에선 단일 top pick(Exact)보다 **Top-k + 에스컬레이션 정확도**가 더 신뢰할 지표.
