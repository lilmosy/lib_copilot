## 2026-07-15 — 청구기호 직접 크롤링 실증 ✅

**질문:** 제목→책→청구기호 클릭 3단계 없이, 청구기호 하나로 바로 그 번호대 전체를 긁을 수 있나?

**결과: 가능.** 검색 축 `st=FRNT`(청구기호/분류기호)를 쓴다.
```
GET https://library.sogang.ac.kr/search/toc/result?st=FRNT&si=12&q=<청구기호>&pn=<페이지>
예) q=" 891.733" → 총 750건, 10건/페이지, pn 페이지네이션 정상
```
- crawler README의 `st=KWRD/TITL/AUTH/PUBL`에 더해 **`st=FRNT`= 청구기호 검색**. (`si=12`)
- 결과 각 항목의 청구기호는 `[대괄호]` 안. 단 제목의 `[and]` 등이 섞일 수 있어 "숫자.숫자+저자기호" 패턴 필터 권장.

**함정 — 봇 검증(`__verified_refresh`) 우회법:**
크롤러로 바로 치면 302 무한 리다이렉트. 2단계로 우회:
1. 1차 요청 → 302 + `Set-Cookie: JSESSIONID, verified`(Max-Age=300s=5분) 수령
2. 같은 URL에 `&__verified_refresh=1` 붙여 재요청(쿠키 포함) → 200 + 결과
- `requests.Session()`이 쿠키 자동 반송. **`verified` 5분 만료** → 장시간 크롤링 시 주기적 재발급 필요.

**함의:** 이제 "청구기호 키로 DB 구축"이 가능. 시나리오의 각 케이스에서 사서가 검색한 청구기호대(예: 720.2, 720.105)를 그대로 긁어 `collection.json`을 **진짜 로욜라 데이터로 교체**할 수 있다.

## 2026-07-22 — 상세페이지 실측 + db.py(pandas) + Streamlit 데모

**상세페이지에서 뭘 일정하게 얻나 (실측, n=60 무작위)**
`/search/detail/{id}`의 `table.profiletable` th/td를 세었다. id는 목록페이지 제목 링크에서 이미 뽑아 저장돼 있어(`BIBID_RE`) URL 조립만 하면 된다.

|---|---|---|---|---|
| 자료유형·서명저자·발행·형태·언어 | 98% | | 서지주기 | 40% |
| 개인저자 | 91% | | 일반주기 | 30% |
| 분류기호 | 83% | | 원서명 | 6% |
| **ISBN** | **75%** | | **내용주기** | **3%** |
| **일반주제명** | **58%** | | 총서명 | 43% |

- **목차·초록은 0%.** `<!-- //목차정보 -->` 마커만 남고 내용 블록이 통째로 비어 있다. 2020~2026년 국내 신간 12권도 전부 없음. → **서강 쪽에서 목차 기대는 접는다.**

- ⚠️ 표본 52~60건이 모두 12케이스 번호대에서 뽑혀 장서 전체 대표성은 아님.

**결론: 목차·책소개는 국중 ISBN 서지정보 API(`/seoji/SearchApi.do`)가 유일 경로.**
- 가이드의 "일반검색/상세검색"은 **같은 API의 요청 방식 2가지**이고 출력 필드(2.3, 28개)는 공통 — 거기엔 목차·책소개·DDC가 없다. 상세검색의 `toc_keyword`는 목차로 *검색*하는 것이지 목차를 *받는* 게 아니다.
- ISBN API에는 `BOOK_TB_CNT`(목차)·`BOOK_INTRODUCTION`(책소개)·`KDC`·`DDC`·`SUBJECT`가 있다. 필수 파라미터는 cert_key/result_style/page_no/page_size뿐, 검색키는 `isbn` 하나면 됨.
- ⚠️ 키 발급 후 **1순위 확인**: (a) `BOOK_TB_CNT`가 2021년 3월 이후 신간에도 오나(27~29번 URL 필드엔 "21년 2월까지" 단서가 있으나 33~35번 본문 필드엔 없음) (b) 국내서 `DDC` 채움 비율 (국내는 KDC 위주일 수 있음).
- 인증키가 자료검색(`key`)과 ISBN 서지정보(`cert_key`)로 별도인지 미확인.

**케이스11 재검증 — recall 실패의 진짜 원인 (중요)**
케이스11 입력은 `title/author/082`뿐이고 부제·키워드가 없다. 그래서 토큰 0매칭이었다.
그런데 키워드 `["부동산","투자"]`만 넣어주면 분포에서 **332.6324가 2권으로 잡힌다 = review_final(교열 최종 정답).**
→ **병목은 검색 방식(토큰 vs 임베딩)이 아니라 입력 메타데이터 부재였다.** "부린 왕자"라는 조어에서 임베딩이 '부동산'을 끌어낼 수는 없다. 목차·책소개가 있어야 한다.
→ **V2 임베딩보다 국중 API(입력 보강)가 우선.** 빌드 순서 수정.
(단, 키워드는 내가 손으로 넣은 것 — 국중 API가 자동으로 채워줄 자리임을 보인 것이지 점수가 아님.)

**db.py 추가 (pandas, 저장소 접근의 유일한 창구)**
- 왜 pandas인가: 1682행≈1MB로 메모리에 다 올라가 인덱스가 필요 없고, 필드가 책마다 달라(라벨 23종, 3~98%) 고정 컬럼 테이블과 안 맞는다. SQLite가 주는 이점(디스크 상주·인덱스·동시 쓰기·고정 스키마)이 지금 하나도 필요 없다. 실질 쓸모는 ①분포 groupby ②Streamlit 표, 둘뿐.
- **DataFrame은 db.py 밖으로 안 나간다.** 밖으로는 `CollectionBook`/`DistBucket`만. → 내부를 SQLite로 갈아끼워도 부르는 쪽 불변(retrieve_token/embed가 RetrieveResult를 공유하는 것과 같은 패턴).
- 갈아탈 기준: 10만 행 초과 / 동시 쓰기 발생 / 앱 시작 1초 초과.
- **`retrieve_token.py`는 건드리지 않았다.** 골든셋 점수(Exact33/Prefix67/Topk67)가 왜 변했는지 알 수 없게 되므로 평가 경로는 동결.
- `data/collection.json`은 분리하지 않고 레코드 안에 `detail` 키로 중첩하기로 결정(스키마를 못 정하는데 파일만 나누면 조인만 는다).

**app.py (Streamlit) — 업무 3단계를 화면으로**
1. 원제 검색 → 기존 서지 승계(결정론적) / 2. 082 → 번호의 '의미' 확인 / 3. 제목 → 분포 → 상위 N → LLM 정합성 → HITL(YES/NO/직접입력).
- 3단계 분포 버킷을 `RetrieveResult`로 옮겨 담는 어댑터를 둬서 **classify.py를 한 줄도 안 고치고 재사용**(LLM 호출은 여전히 한 곳).
- 검증: 케이스5 분포가 `retrieve_token`과 정확히 일치(720.2:3, 711.4:1, 총 4건). 서버 기동 HTTP 200.

## 2026-07-22 — 토큰 규칙 단일화 + 082 기각 상태(C안) + 결정 로그

**토큰 규칙 복제 제거**
`retrieve_token`(평가 경로)과 `db`(데모 경로)에 토큰 규칙이 복사돼 있었다 → `src/tokens.py` 하나로 통합. `MIN_OVERLAP`의 주인은 `config.py`(tokens는 재수출만). 자르기·정렬·082 강제 포함도 두 경로 동일하게 맞춤.

**082 신분 문제 (사용자 지적)**
2단계에서 082를 기각했는데 3단계 후보 맨 앞에 082가 다시 올라오는 건 업무 절차와 어긋난다. 케이스5 `human_trace`가 증거: ①082 기각 → ②분포 → ③재선택 순서로, 082는 ②에 들어갈 때 이미 탈락 상태다.
- 왜 `retrieve_token`에선 맞았나: 거긴 2단계가 따로 없어 한 방에 판단하므로 082가 후보에 있어야 '기각 근거'가 나온다. **같은 코드가 단계 구분 유무에 따라 맞고 틀림.**
- 채택: **C안** — 후보에는 남기되 (a) 1순위 우선권 제거 (b) "사서가 기각함"을 LLM에 명시.
  - 빼지 않는 이유 셋: 기각 근거를 산출물에 남겨야 함 / 사서가 틀렸을 수 있어 LLM 반박 여지 필요(§5 단일 정답 강제 금지) / 스키마에 `CandidateSource.REJECTED_082` 자리가 이미 있음.
- 082 상태 3종: 미판정(prior 대접, 맨 앞) / 정합함(채택·종료) / 기각(후보엔 남되 개수순으로 밀림).
- 검증: `prior_rejected=False → [720.105, 720.2, 711.4]`, `True → [720.2, 711.4, 720.105]`.

**classify.py에 `human_notes` 선택 인자 추가**
사서의 앞 단계 판정을 프롬프트에 주입. **기본 None → 평가 경로 프롬프트는 한 글자도 안 변함**(검증: None일 때 블록 미삽입, 길이차 0). 평가 경로 동결 유지.

**2단계에 판정 UI 추가**
전에는 조회만 하고 끝났다(업무상 가장 중요한 갈림길인데). 이제 [정합함 → 채택·종료] / [정합 안 함 → 3단계] 버튼.

**결정 로그 `runs/decisions_YYYYMMDD.jsonl`**
사서 최종 결정(accept/reject/manual) + LLM 1순위 + 에스컬레이션 신호를 append. 쌓이면 'LLM 1순위와 사서 최종이 얼마나 갈리나'를 실측 가능 → 골든셋 3건 외 실사용 데이터.
- ⚠️ **`collection.json`에는 절대 되쓰지 않는다.** 결정 결과가 DB에 들어가면 평가 시 정답이 코퍼스에 섞여 누출이 된다(§8, 케이스3이 이미 그 문제). 앱 전체가 읽기 전용이며 `data/`는 7/15 크롤 이후 수정 없음.

**미해결(구조적):** 3단계가 종합서지 없이 본교 DB로 대체 중이라 2·3단계가 같은 DB를 본다 → 단계 구분이 흐려진다. 국중 seoji API로 3단계 분포를 채울 수 있는지가 관건(단 DDC 채움 여부 미확인).
