# lib_copilot 설계서 (Ver 1)

> 도서관 청구기호(852 ▼h) 부여를 돕는 **사서-AI 협업 에이전트**.
> 실제 사서의 업무 흐름을 그대로 코드 로직으로 옮기고, 사람이 판단해야 하는 지점(HITL)만 남긴다.
> 기반: 교수님의 [`sogang_renAIssance`](../../../sogang_renAIssance) 설계를 참조하되, **실제 업무 프로세스**에 맞춰 다시 설계함.

**요점 한눈에**
- 입력(서지정보) → **① retrieve(후보 번호 넓게) → ② classify(LLM, 주제·급으로 판단)** → ▼h 후보 + 근거 + HITL.
- 난이도(하/중/상)는 라벨이 아니라 **에스컬레이션 신호의 강도**. 애매하면 사람에게 넘긴다.
- 성능은 **골든셋(정답 숨긴 시험)**으로 채점. 개념: `데이터 계약(§3)`, `모듈(§4)`, `평가(§8)`.

---

## 0. 문제 선언

현재 청구기호(특히 852의 ▼h)를 정하는 과정은 **repetitive하고 비일관적**이다.

- **repetitive** → 자동화한다.
- **비일관적(사서 암묵지)** → 일관된 로직으로 대체하고, 사람은 **판단이 갈리는 지점에만** 개입한다.

목표: 시간 단축 + 암묵지(tacit)를 분명한 규칙으로 만들기.

### 왜 ▼h만인가 (스코프)
852의 다른 하위요소는 규칙/근사로 처리 가능하고, 진짜 "판단"이 갈리는 곳은 ▼h(분류기호)다.
그래서 Ver 1의 지적 난이도는 **▼h 결정 한 곳에 집중**한다. (▼i 도서기호·▼m 부가기호·852 조립은 이후 과제)

## 1. 시뮬레이션 전제 (중요)

실제 도서관 시스템에 붙지 않는다. **시나리오 기반 시뮬레이션**으로 개발·평가한다.

- **정답셋(ground truth):** 고은님이 근무 중 수기 기록한 케이스 분석(12개) → `data/scenarios/`에 골든셋 JSON으로 정규화.
- **DB 가정:** 업체 082를 실시간으로 받는 파이프는 안 만든다. 대신 **"업무 중 마주치는 책들"**(청구기호·제목 검색으로 떠오르는 책)을 모은 DB(`collection.json`)가 있다고 가정하고, 여기서 정답에 수렴시킨다.
- **개발 방식:** 사례 하나를 end-to-end로 굴리며 시행착오 → 평가를 고도화. 하/중/상을 딱 나누는 완벽한 기준은 없다.

## 2. 시스템 개요 · 업무 흐름

### 2.1 한눈에 보는 입력 → 출력

```
[입력] 신규 도서 서지정보 (BookInput)
   예) 제목「가장 인간적인 도시」 · 부제「…AI시대 건축」 · 저자 정현재 · 082=720.105
    │
    ▼  ① 1차 retrieve  (recall 넓게)   → 후보 청구기호 + 각 번호대 샘플 책
    │     예) 후보 {720.105(082), 720.2, 711.4} + 각 번호대 책 제목들
    ▼  ② 2차 classify (Claude, 판단)   → ▼h 후보 + 근거 + 에스컬레이션
    │
[출력] ClassificationResult  (구조화 JSON, 자유 텍스트 한 줄 아님)
   예) [1] 720.2 도시·건축(교양) 0.60  [2] 720.105 …  → escalate: needs_toc(HITL)
```

- **입력** = 서지정보 JSON 한 건. **출력** = ▼h 후보 목록 + 근거 + HITL 여부(구조화). 계약은 §3.
- 실행: `python src/run.py [골든셋경로]` → 콘솔 출력 + `output/`에 json 덤프.

### 2.1.1 입구가 둘이다 — 평가 경로 / 데모 경로

같은 판단부(`classify.py`)를 두 입구가 공유한다. **차이는 "누가 후보를 만드느냐" 하나뿐이다.**

```
【평가 경로】 run.py → pipeline.py → retrieve_token ─┐
                                                    ├─→ classify.py → Claude
【데모 경로】 app.py → db.py → (어댑터) ─────────────┘
```

| | `retrieve_token` 경로 | `db.py` 경로 |
|---|---|---|
| 입력 | `BookInput` **한 번에** | 사서가 **단계마다 다른 것** (1단계 원제 / 2단계 082 / 3단계 제목) |
| 후보 생성 | **자동** — DB 전체를 토큰 겹침으로 훑음 | **사람이 좁힘** — 분포 보고 상위 N 선택 |
| 사람 개입 | 없음 | 흐름의 일부(HITL) |
| 용도 | **골든셋 채점** (같은 입력 → 같은 후보 → 점수 비교) | **업무 재현 + 시연** |

> `db.py`에는 후보 생성 로직의 **일부만** 있다: 토큰 매칭·번호별 집계는 `retrieve_token`과 동일(`tokens.py` 공유),
> 082 강제 포함·`RetrieveResult` 조립은 `app.py`의 어댑터가 맡는다. `pipeline.py`의 배선 역할은 화면 단계 흐름이 대신한다.

**두 경로의 동작은 일치시킨다**(토큰 규칙·082 강제 포함·정렬·상한). 어긋나면 "데모에서 본 것"과 "평가 점수"가 다른 것을 가리키게 된다.
검증: 케이스 3·5·11에서 두 경로의 후보 목록이 완전 일치.

### 2.2 실제 사서 업무 흐름 (이걸 이식한다)

사서는 082(또는 후보번호)를 **본교(로욜라)에서 확인**하고, **더 큰 풀(종합서지)에서 분포**를 본 뒤, **다시 본교로 되돌려** 적합을 확인한다:

```
082 확인
 ① 청구기호 검색 → 본교서지 : 082(또는 후보번호)가 본교에서 뭘 의미하나 / align 하나
 ② 제목 검색   → 종합서지 : 더 큰 풀(타학교)에서 유사 도서가 어느 번호에 몰리나 (분포)
 ③ 청구기호 검색 → 본교서지 : ②의 후보 번호를 다시 본교로 가져와 의미·적합 확인
 ④ 확정
```

### 2.3 우리 Ver1 근사 (②+③을 본교 하나로 합침)

로욜라는 **본교서지**라 종합서지가 없다. 그런데 사서 결론(부록2): *"종합서지는 후보 제안만 할 뿐 판단 권한 없음. 후보 제안은 본교 제목·키워드 검색으로도 됨 = 종합서지 꼭 필요 없음."* → **②의 종합서지를 본교 DB 제목검색으로 대체**해 ②+③을 하나로 합친다:

```
제목 검색 → 본교 DB → 유사 도서들의 청구기호 = 후보들
   → 각 후보의 '의미'는 이미 본교 기준 (내 DB니까)
   → LLM이 분포(개수) + 의미(주제 적합) + 규칙으로 최종 판단
```

> ⚠️ **본교-only의 한계:** 본교에 책이 있는 번호만 후보로 낼 수 있다(케이스5의 720.563은 본교 0권이라 후보에서 빠짐 — 대개 정합이나, 새 주제로 본교가 빈약하면 후보를 놓쳐 상 케이스/HITL로 감). 담당 사서 확인 권장.

### 2.4 funnel 구조 (recall → precision)

082 유무로 진입 경로가 갈리고(082 있음 → 청구기호 검색부터 / 없음 → 제목 검색부터), 에이전트는 이를 **말로 설명하며** 진행한다(XAI). 전체는 **깔때기**다:

```
1682건 (collection.json = 본교 장서 풀)
   │  ① 1차 retrieve (recall, 넓게·싸게)          RetrieveResult
   │     제목/부제/키워드로 후보 번호 건짐 (+082 prior). 각 번호대 샘플 책 첨부.
   ▼
후보 번호 ~N개 (+ 각 번호대 대표 책들)
   │  ② 2차 classify (precision, LLM)             ClassificationResult
   │     샘플 책으로 '주제·급'을 파악해 ▼h 판단. 개수는 보조. 애매하면 escalate.
   ▼
▼h 후보 1~3개 + 근거 (+ 필요시 HITL)
```

- **1차는 정확할 필요 없다(recall 우선).** threshold를 칼같이 정하지 않고 넉넉히 건진다. 정답 판단은 2차.
- **1차 = 두 구현, 같은 계약(RetrieveResult).** 내부만 다르고 배선/2차는 불변. `config.RETRIEVER`로 스위치, evaluate로 비교.
  - `retrieve_token` (V1): 제목 토큰 겹침 ≥ `MIN_OVERLAP`. 싸고 단순. 제목 안 겹치면 못 잡음.
  - `retrieve_embed` (V2): 임베딩 의미 유사("AI시대"↔"인공지능시대"). 백엔드 결정 후 구현.
- **LLM 호출은 classify 한 곳에 집중.** 제목 유사도는 후보를 건지는 그물일 뿐, 결정은 2차가 주제로.

> **주제 우선 · 개수 보조 (원칙).** 후보 번호의 '개수'는 결정자가 아니다: (a) DB는 번호당 최대 50건 **샘플**이라 개수 부정확(잘림 → `crawl_coverage.json`), (b) 개수 최다가 정답 아닌 케이스 존재(케이스11: 179.9 최다이나 오답). → 2차는 shelf_sample로 **주제 적합**을 최우선 판단.

## 3. 데이터 계약 (schema) — `src/schema.py`

단계 간에 오가는 타입. 내부 구현이 바뀌어도 이 계약은 고정한다. (**최우선 확정 대상**)

### 3.1 BookInput (입력)
```jsonc
{
  "title": "가장 인간적인 도시",
  "subtitle": "실리콘밸리의 젊은 건축가가 안내하는 AI시대 건축",
  "author": "정현재",
  "ddc_082": "720.105",     // 선택: 업체 DDC(prior). 없으면 null
  "is_translation": false,
  "original_title": null, "original_lang": null,
  "keywords": [], "toc": [], "description": null   // 상 케이스에서만 채워질 수 있음
}
```
> ⚠️ `keywords`의 출처는 미정(열린 질문 §7). 현재 골든셋 keywords는 셋업 시 수기로 넣은 것 → analyze 단계(LLM 키워드 추출)로 정식화 필요.

### 3.2 RetrieveResult (① 1차 산출물)
```jsonc
{
  "retriever": "token",              // token | embed
  "query_terms": ["도시","건축", ...],
  "messages": ["082(720.105) 있음 → …"],   // 진행 멘트(XAI)
  "total_title_hits": 4,
  "candidates": [                    // 후보 청구기호 + 근거
    { "ddc_h": "720.2", "is_082_prior": false,
      "title_hit_count": 3,          // 제목 매칭이 이 번호 가리킨 수
      "shelf_count": 50,             // DB 이 번호대 책 수(상한 샘플 → 부정확)
      "title_hits": ["…"], "shelf_sample": ["…"] }  // 주제 파악용 책 제목
  ]
}
```

### 3.3 ClassificationResult (② 2차 산출물 = 출력)
```jsonc
{
  "candidates": [                    // 적합도 내림차순
    { "h": "720.2", "label": "도시·건축(교양)", "confidence": 0.60,
      "source": "title_match",       // 082_kept | 082_rejected | title_match | inherited
      "reasoning": "…대중 교양서라 교양 서가…", "similar_refs": ["…"] }
  ],
  "ambiguity": 0.3,                  // 0(명확)~1(모호)
  "escalate": true,                  // HITL 필요?
  "signals": ["needs_toc"],          // cross_major | needs_toc | scattered_dist | org_disagree
  "notes": "종합 판단 근거 요약"
}
```

### 3.4 저장소 계약 — `CollectionBook` / `DistBucket`

`db.py`가 밖으로 내보내는 타입. **DataFrame은 `db.py` 밖으로 나가지 않는다**(§4 참조).

```jsonc
// CollectionBook — 본교 장서 한 건 (이미 청구기호가 붙은 책)
{
  "id": "CATTOC000000692092", "title": "...", "author": "...",
  "call_number": "174.4 B419", "ddc_h": "174.4",
  "detail": {                       // 상세페이지 th/td 원자료. 크롤 전에는 {}
    "일반주제명": "Capitalism -- Moral and ethical aspects",
    "ISBN": "9788964600559", "원서명": "..."
  }
}

// DistBucket — 제목 검색 분포의 한 칸 ('720.2가 6권')
{ "ddc_h": "720.2", "count": 6, "books": [...], "shelf_sample": [...], "shelf_count": 50 }
```

`detail`은 **필드를 미리 고르지 않는다.** 라벨이 23종 이상이고 책마다 다르며(3~98%), 재크롤이 3시간이라
나중에 필요해질 필드를 위해 원자료를 통째로 담는다. 해석은 파이프라인 쪽에서 한다(§11.1).

## 4. 구성 모듈 상세

> 경계 = "정해진 입력 → 정해진 출력"의 순수 함수. 내부만 갈아끼운다. (교수님 설계 사상 계승)

| 모듈 | 역할 | 입력 → 출력 | 구현 요점 |
|---|---|---|---|
| `crawler/callno_crawler.py` | DB 수집(넓게) | 청구기호 리스트 → collection.json/csv + coverage | 검색축 `st=FRNT`, `__verified_refresh` 쿠키 우회(Session), 번호당 5p=50건 상한, 잘림 로그 |
| `crawler/detail_crawler.py` | DB 수집(깊게) | collection.json → 각 건에 `detail` 추가 | 목록 id로 `/search/detail/{id}` 조립 → `table.profiletable` th/td **통째로**. 건당 ~7초(전체 3h) → **50건마다 중간 저장**, `--resume`/`--limit` |
| `tokens.py` | 토큰 규칙 | 문자열 → 토큰 집합 | `STOP`·`tokens()`·`overlaps()`. **`retrieve_token`과 `db`가 공유**(복제 금지). `MIN_OVERLAP`의 주인은 `config.py` |
| `retrieve_token.py` (V1) | 1차 recall | BookInput → RetrieveResult | 제목+부제+키워드 토큰화, DB 제목과 겹침 ≥`MIN_OVERLAP`(컷 없이 넓게), 걸린 책의 ddc_h를 번호별 grouping, 082 prior 강제 포함 |
| `retrieve_embed.py` (V2) | 1차 recall | BookInput → RetrieveResult | 임베딩 의미 유사(백엔드 미정). 조어·동의어("AI시대↔인공지능시대") 잡기 목표 |
| `db.py` | 저장소 창구 | 질의 → CollectionBook / DistBucket | **내부만 pandas, DataFrame은 밖으로 안 나감.** 조회 3종(원제/청구기호/제목분포). 저장 방식을 바꿔도 부르는 쪽 불변 |
| `classify.py` ★ | 2차 판단 | BookInput + RetrieveResult (+ `human_notes`) → ClassificationResult | **유일한 LLM(Claude, `output_format` 강제).** shelf_sample로 주제+**급/장르** 판단, 개수 보조, 도메인 규칙 주입, 에스컬레이션 신호 판정. `human_notes`는 데모 전용(기본 None → 평가 프롬프트 불변) |
| `pipeline.py` | 배선 | BookInput → PipelineOutput | `config.RETRIEVER`로 1차 스위치 → 2차 연결 |
| `run.py` | 실행기(CLI) | 골든셋 경로 → 콘솔 + `output/*.json` | 케이스 실행, 1차/2차/합본 3종 덤프 |
| `app.py` | 데모 UI | 사서 조작 → 화면 + `runs/*.jsonl` | Streamlit. 업무 3단계를 화면으로(§2.2). 분포 버킷 → `RetrieveResult` 어댑터로 classify 재사용 |
| `evaluate.py` | 평가 하니스 | `scenarios/*.json` → Exact/Prefix/Top-k | 정답 숨기고 예측 후 대조. 검색기별 비교. gold 두 층(작성자/교열) |
| `schema.py` / `config.py` | 계약 / 설정 | — | §3 타입 정의 / 모델·경로·RETRIEVER·MIN_OVERLAP·SHELF_SAMPLE |

### 4.1 왜 pandas인가 (SQLite가 아니라)

`db.py` 내부 선택. **SQLite가 주는 이점이 지금 하나도 필요 없어서**다 — 짧아서가 아니다.

| SQLite의 존재 이유 | 우리 상황 |
|---|---|
| 데이터가 메모리에 안 들어감 | 1682행 ≈ 1MB. 전부 올라감 |
| 인덱스로 조회 가속 | 전체 스캔이 밀리초 |
| 여러 프로세스 동시 쓰기 | 쓰는 건 크롤러 하나, 그것도 가끔 |
| 고정 스키마 보장 | 필드가 책마다 다름 — **오히려 방해** |

pandas의 실질 쓸모도 둘뿐이다: ① 분포 `groupby` ② 화면 표. 나머지는 순수 파이썬으로도 된다.
**갈아탈 기준:** 10만 행 초과 / 동시 쓰기 발생 / 앱 시작 1초 초과 → 그때 `db.py` 내부만 교체.

원본은 어디까지나 **JSON**이다(크롤러가 쓰고, git에 남고, 사람이 읽는다). pandas는 런타임 뷰일 뿐.

## 5. 난이도 = 고정 라벨이 아니라 "에스컬레이션 신호의 강도"

케이스 분석의 하/중/상은 **딱 나뉘는 정답 라벨이 아니다.** 부록1도 상(上)을 **트리거 신호**로 정의한다. 시스템은 난이도를 *입력으로 받지 않고* **검색 결과의 모양을 보고 스스로 판정**한다.

### 에스컬레이션 신호 (HITL 트리거)
1. 후보가 **성격 다른 대분류를 넘나듦** (문학 800 vs 부동산 330 vs 성찰 170)
2. **제목·부제·저자만으로 초점 판별 불가** → 목차/책소개 필요
3. **후보 분포가 흩어져 수렴 안 함**
4. **기관 간 분류 상이** (교보·알라딘)

| 등급 | 성격 | 에이전트 동작 |
|---|---|---|
| 하 | 규칙·서지 검색 명확히 수렴 | 자동 처리 |
| 중 | 082 기각/인접 선택 필요하나 서지 의미로 방어 가능 | 자동 처리 + 근거, 저신뢰 시 플래그 |
| 상 | 실질적 다의성, 사서마다 갈림 | **HITL 필수** — 단일 정답 강제 금지, 후보+근거형 |

> ▼h는 두 층위 정답을 가짐: `작성자 확정` vs `교열/최종`. 상 케이스는 둘이 다르다(뒤집힘).
> → 상 케이스 평가는 "정답 하나 맞히기"가 아니라 **Top-k + 에스컬레이션 정확도**로 본다.

### 5.1 082의 세 가지 신분 (데모 경로)

한 방에 판단하는 평가 경로와 달리, 데모는 2단계(082 확인)와 3단계(분포)가 **나뉘어 있다.**
그래서 082가 3단계에 들어갈 때의 신분이 달라진다:

| 2단계 판정 | 3단계에서 082는 | 근거 |
|---|---|---|
| **미판정** (2단계 안 거침) | 후보 **맨 앞** (prior 대접) | 082 없는 책도 있어 3단계부터 시작 가능 |
| **정합함** | 3단계에 안 감 (채택·종료) | 사서 업무 그대로 |
| **기각** | 후보에 **남되 1순위 자리는 뺌** + LLM에 "사서가 기각함" 명시 | 아래 |

**기각된 082를 후보에서 지우지 않는 이유 셋:**
1. 최종 산출물에 *"082는 720.105였고 이래서 기각했다"*가 남아야 업무 기록으로 쓸모가 있다.
2. 사서가 틀렸을 수 있다 — 분포까지 본 LLM이 반박할 여지를 남긴다(단일 정답 강제 금지).
3. 스키마에 `CandidateSource.REJECTED_082` 자리가 이미 있다. 이제 그 값은 **사서가 실제로 누른 결과**다.

> 근거: 케이스5 `human_trace`가 ①082 기각 → ②분포 → ③재선택 순서다. 082는 ②에 들어갈 때 이미 탈락 상태다.
> 같은 코드(082 강제 포함)가 평가 경로에선 맞고 데모 경로에선 틀린 이유가 여기 있다 — **단계 구분 유무**가 가른다.

## 6. 도메인 규칙 (케이스 분석에서 추출 → classify에 주입)

- **본교 서지 최우선.** 기존 서지/판본 있으면 그 청구기호 **그대로 승계**(결정론적). 타학교가 달라도 무시.
- **번역서:** 원제 검색 → 기존 원서/번역서 청구기호 **우선**, ▼m에 `K`. 외국인 저자는 항상 전거 생성 → 전거 없음 = 책 없음.
- **082는 prior.** 주제와 align 안 되면 기각하고 서지 의미/분포로 재선택.
- **종합서지는 후보 제안만.** 판단 권한 없음(본교 제목·키워드 검색으로 대체 가능).

## 7. 설계 질문 — 결정 현황

케이스5를 굴리며 정한 것들. 근거는 `devlog.md`, 관찰은 `output/`.

- **책 정보를 어느 정도 받나?** → `BookInput`(제목·부제·저자·082·키워드·목차·책소개). 목차/책소개는 상 케이스만.
- **제목 "비슷"을 어떻게? (하드코딩 vs LLM vs 임베딩)** → **funnel로 우회**(1차 recall 넓게, 판단은 2차). 토큰(V1)/임베딩(V2) 두 버전 → 평가로 결정. ⚠️ 한국어 토큰은 조사가 붙어("건축가가"≠"건축") 약함 → V2 근거.
- **LLM이 주제를 무엇으로?** → 후보 번호대 **shelf_sample로 주제·급 파악**. ⚠️ shelf_sample이 편향되면 판단도 편향 → 대표성이 지렛대(케이스5, `output/case05_*_analysis.md`).
- **개수를 신뢰?** → 아니오. **주제 우선·개수 보조**(§2.4).
- **저장소를 뭘로?** → **JSON 원본 + pandas 런타임**(§4.1). SQLite는 아직 아님. 갈아탈 기준 명시됨.
- **키워드 출처?** → ⚠️ **미해결. 데모에서 실제로 터졌다.**

#### 7.1 키워드 문제 — 골든셋의 숨은 전제

골든셋 입력에는 `keywords`가 **미리 채워져 있다**(케이스5: `["도시","건축","AI시대","공간"]`).
`retrieve_token`이 제목+부제+키워드를 합쳐 토큰화하므로, 그 덕에 매칭이 성립했다.
**그런데 이 키워드는 사람이 손으로 넣은 것이고, 실전에는 없다.**

Streamlit 데모가 이 전제를 드러냈다 — 사서가 제목만 입력하면 대부분 0건이 나온다:

| 입력 | 겹침≥2 | 겹침≥1 |
|---|---|---|
| 「가장 인간적인 도시」 제목만 | **0권** | 16권 / 6종 |
| + 부제·키워드 | 4권 / 2종 | — |
| 「부린 왕자」 제목만 | **0권** | 1권 |

해법 셋, 결국 하나로 수렴한다:

| | 방법 | 한계 |
|---|---|---|
| A | 사서가 키워드 직접 입력 (현재) | 실전 부담 + **사서마다 달라지면 프로젝트가 없애려던 비일관성이 재현됨** |
| B | `analyze` 단계 — LLM이 제목·부제에서 추출 | 「부린 왕자」처럼 제목에 정보가 없으면 무력 |
| C | **국중 API로 목차·책소개 → 키워드 자동 생성**(§11.2) | 근본 해법 |

실증: 케이스11에 키워드 `["부동산","투자"]`만 넣으면 분포에서 **332.6324(교열 최종 정답)가 바로 뜬다.**
→ **병목은 검색 방식(토큰 vs 임베딩)이 아니라 입력 메타데이터 부재였다.** 조어 제목에서 임베딩이
'부동산'을 끌어낼 수는 없다. 목차·책소개가 있어야 한다. **⇒ V2 임베딩보다 §11.2가 우선.**

> 완화책(시연용): 3단계에 **골든셋 예제 불러오기** 드롭다운. 케이스 3/5/11을 클릭 한 번으로 재현한다.
> 실전 입력 문제와는 별개이며, 근본 해결이 아니다.
>
> 완화책(검색): 화면의 **검색 강도**(=`min_overlap`) 조절. 기본 2(평가와 동일), 0건이면 1로 낮춰 넓게 훑는다.
> `config.MIN_OVERLAP` 기본값은 건드리지 않아 평가 점수는 불변. §2.4의 "1차는 recall 우선"과 정합.

## 8. 평가 (evaluate)

- **정량:** 골든셋의 교열 최종 ▼h 대비
  - **Exact**(완전 일치) — 엄격, 하/중 위주
  - **Prefix**(앞 3자리/대분류 일치) — **1차 우선 지표** (720.2↔720.105도 "720"은 맞음)
  - **Top-k**(후보 안에 정답 포함) — 상 위주
  - baseline: 검색 분포 최빈 ▼h(다수결) 대비 LLM+규칙 기여
- **정성:** 상 케이스에서 복수 후보·근거 타당성, 에스컬레이션 판정이 맞았나.
- **비결정성:** 애매 케이스는 top pick이 실행마다 흔들림(케이스5) → **Exact 대신 Top-k+에스컬레이션**이 신뢰 지표. (다중 샘플/저온도 고려)

### 8.1 결정 로그 — `runs/decisions_YYYYMMDD.jsonl`

데모에서 사서가 내린 최종 결정을 한 줄씩 append 한다(골든셋 3건 밖의 **실사용 데이터**).

```jsonc
{"ts":"...","step":3,"title":"가장 인간적인 도시","ddc_082":"720.105","rejected":true,
 "dist":{"720.2":3,"711.4":1},"llm_top":"720.2","escalated":true,"signals":["needs_toc"],
 "action":"accept","final":"720.2","agreed":true}
```

`agreed`가 쌓이면 **"LLM 1순위와 사서 최종이 얼마나 갈리나"**를 실측할 수 있다.

> ⚠️ **`collection.json`에는 절대 되쓰지 않는다.** 실제 업무 감각으로는 "확정된 책을 장서에 추가"가
> 자연스럽지만, 여기서는 아래 누출을 **매번 새로 만드는** 짓이다. 앱 전체가 읽기 전용이다.
> 굳이 남기려면 별도 파일(`data/decided.json`)로 분리하고 평가 시엔 읽지 않는다.

### ⚠️ 데이터 누출 & 홀드아웃 (미구현, 다음 과제)
정답 **라벨**(expected)은 코드로 깔끔히 숨겨진다 — 파이프라인은 `input`만 받고 `BookInput`엔 정답 필드가 없다. **그러나** DB를 같은 도서관(로욜라)에서 크롤해서, 시나리오 입력 도서가 이미 장서에 있으면 **정답째로 코퍼스에 들어온다**(간접 누출). 실제로 케이스3「향신료의 역사」는 DB에 641.338309로 등재돼 있어 retrieve가 그 책을 그대로 찾아 "판단" 아닌 "승계"로 맞힘 → **점수 부풀림**. (케이스5·11 입력은 DB에 없어 누출 없음 → 진짜 난이도가 드러남.)
→ 공정 평가엔 **홀드아웃**(평가 대상 책을 DB에서 제외) 필요.

## 11. 외부 데이터 소스

### 11.1 서강 상세페이지 — 실측 결과

`/search/detail/{id}`의 `table.profiletable`. id는 목록페이지 제목 링크에서 이미 추출돼 있어 URL 조립만 하면 된다.

| 필드 | 출현율 | 쓸모 | | 필드 | 출현율 |
|---|---|---|---|---|---|
| 자료유형·서명저자·발행·형태·언어 | 98% | 기본 | | 서지주기 | 40% |
| 개인저자 | 91% | 전거 | | 일반주기 | 30% |
| 분류기호 | 83% | (이미 보유) | | 총서명 | 43% |
| **ISBN** | **75%** | ★ 국중 API 조인 키 | | **원서명** | **6%** |
| **일반주제명** | **58%** | ★ `keywords` 구멍 | | **내용주기** | **3%** |

(무작위 60건 표본. ⚠️ 12케이스 번호대에서만 뽑아 장서 전체 대표성은 아님)

- **목차·초록은 0%.** `<!-- //목차정보 -->` 마커만 남고 내용 블록이 통째로 비어 있다. 2020~2026년 국내 신간 12권도 전부 없음.
  → **서강에서 목차를 얻는 계획은 폐기.**
- `내용주기`(MARC 505)가 목차에 가장 가깝지만 3%, 그것도 수록작을 나열하는 논문집·단편집뿐. 정작 목차가 필요한 단일 주제 신간엔 안 붙는다.
- 이 크롤의 **최대 수확은 ISBN 75%** — 장서를 국중 API로 역조회할 열쇠다.

### 11.2 국립중앙도서관 Open API — 어느 것을 쓰나

가이드가 헷갈리게 돼 있다. **API는 2개**이고 "일반검색/상세검색"은 그중 하나의 **요청 방식**이다.

```
API ①  자료검색  /NL/search/openApi/search.do
        ├ 2.1 일반검색  kwd=…              ← 요청 방식
        └ 2.2 상세검색  f1=title&v1=…      ← 요청 방식
        └ 2.3 출력 필드(28개) ················ 둘 다 여기로 나옴

API ②  ISBN 서지정보  /seoji/SearchApi.do
        └ 3.2 출력 항목(40개)
```

**"상세검색"은 조건이 상세한 것이지 결과가 상세한 게 아니다.** 출력은 공통이고 거기엔 목차·책소개·DDC가 없다.
(상세검색의 `toc_keyword`는 목차로 *검색*하는 것이지 목차를 *받는* 게 아니다.)

| API | 출력 | 목차 | 책소개 | DDC |
|---|---|---|---|---|
| ① 자료검색 | title_info, author_info, call_no, isbn, kdc_code_1s(**대분류 1자리**) | ✗ | ✗ | ✗ |
| **② ISBN 서지정보** | TITLE, AUTHOR, EA_ISBN, PUBLISHER, **KDC, DDC, SUBJECT, BOOK_TB_CNT, BOOK_INTRODUCTION** | **○** | **○** | **○** |

**⇒ 주력은 API ② 하나.** 필수 파라미터는 `cert_key`/`result_style`/`page_no`/`page_size`뿐이고
검색키는 `isbn` 하나면 된다(입력이 상세할 필요 없음). GET이라 브라우저 주소창으로 바로 테스트된다.

두 용도:
- **(a) 신규 책 `BookInput` 보강** — 목차·책소개 → 키워드. §7.1의 근본 해법
- **(b) 장서 1682건 역조회**(ISBN 75%) — `shelf_sample`을 "제목 목록"에서 "주제를 아는 서가"로

**키 발급 후 1순위 확인 3가지:**
1. `BOOK_TB_CNT`가 2021년 3월 이후 신간에도 오나 (27~29번 URL 필드엔 "21년 2월까지" 단서가 있으나 33~35번 본문 필드엔 없음)
2. 국내서 `DDC` 채움 비율 — 국내 도서관은 KDC 위주라 비어 있을 수 있다
3. 케이스 12권 ISBN 조회 시 목차 확보율

⚠️ 인증키가 API ①(`key`)과 ②(`cert_key`)로 별도 발급인지 미확인.

#### DDC가 비어 있으면?

```
DDC 채워짐?
 ├─ YES → 추가로 가능: seoji `title=` 검색의 DDC 분포를 3단계 "종합서지" 자리에 투입
 │         + 자료검색 `gu2=ddc&guCode2=720.2`로 shelf_sample 보강
 └─ NO  → 위 둘 폐기. 3단계 분포는 계속 본교 DB로. SUBJECT만 keywords 보강에 사용

 ★ 어느 쪽이든 목차·책소개(주력)는 그대로 확보된다
```

**KDC→DDC 변환으로 우회하지 않는다.** 두 체계는 1:1 대응이 아니며(특히 한국사·한국문학),
변환하면 근거 없는 번호를 지어내게 된다(AGENTS.md "안 지어냄"과 충돌).

> ⚠️ **국중은 "분포"가 아니다.** 회의록이 원한 종합서지는 *"여러 대학이 같은 책에 매긴 번호가 어떻게 갈리나"*인데,
> 국중은 기관 하나다. seoji로 얻는 건 *"이 주제 책들이 통상 어느 번호로 나가나"*(출판계 관행)로 성격이 다르다.
> 진짜 기관 간 분포는 RISS/KERIS 종합목록 영역이며 현재 스코프 밖이다.

### 11.3 미해결 — 2·3단계가 같은 우물을 본다

3단계가 종합서지 없이 본교 DB로 대체 중이라(§2.3), **2단계와 3단계가 같은 DB를 조회한다.**
원래 3단계의 의의는 "더 큰 풀에서 남들은 어떻게 하나"인데 지금은 같은 우물을 두 번 들여다보는 셈이다.
§11.2의 (a)가 이 자리를 채울 수 있는지가 관건이고, 그건 DDC 채움 여부에 달렸다.
