# lib_copilot 설계서 (Ver 1)

> 도서관 청구기호(852 ▼h) 부여를 돕는 **사서-AI 협업 에이전트**.
> 실제 사서의 업무 흐름을 그대로 코드 로직으로 옮기고, 사람이 판단해야 하는 지점(HITL)만 남긴다.
> 기반: 교수님의 [`sogang_renAIssance`](../../../sogang_renAIssance) 설계를 참조하되, **실제 업무 프로세스**에 맞춰 다시 설계함.

**요점 한눈에**
- 입력(서지정보) → **① retrieve(후보 번호 넓게) → ② classify(LLM, 주제·급으로 판단)** → ▼h 후보 + 근거 + HITL.
- 난이도(하/중/상)는 라벨이 아니라 **에스컬레이션 신호의 강도**. 애매하면 사람에게 넘긴다.
- 성능은 **골든셋(정답 숨긴 시험)**으로 채점. 개념: `데이터 계약(§3)`, `모듈(§4)`, `평가(§8)`.

---

## 0. 문제 선언

현재 청구기호(특히 852의 ▼h)를 정하는 과정은 **repetitive하고 비일관적**이다.

- **repetitive** → 자동화한다.
- **비일관적(사서 암묵지)** → 일관된 로직으로 대체하고, 사람은 **판단이 갈리는 지점에만** 개입한다.

목표: 시간 단축 + 암묵지(tacit)를 분명한 규칙으로 만들기.

### 왜 ▼h만인가 (스코프)
852의 다른 하위요소는 규칙/근사로 처리 가능하고, 진짜 "판단"이 갈리는 곳은 ▼h(분류기호)다.
그래서 Ver 1의 지적 난이도는 **▼h 결정 한 곳에 집중**한다. (▼i 도서기호·▼m 부가기호·852 조립은 이후 과제)

## 1. 시뮬레이션 전제 (중요)

실제 도서관 시스템에 붙지 않는다. **시나리오 기반 시뮬레이션**으로 개발·평가한다.

- **정답셋(ground truth):** 고은님이 근무 중 수기 기록한 케이스 분석(12개) → `data/scenarios/`에 골든셋 JSON으로 정규화.
- **DB 가정:** 업체 082를 실시간으로 받는 파이프는 안 만든다. 대신 **"업무 중 마주치는 책들"**(청구기호·제목 검색으로 떠오르는 책)을 모은 DB(`collection.json`)가 있다고 가정하고, 여기서 정답에 수렴시킨다.
- **개발 방식:** 사례 하나를 end-to-end로 굴리며 시행착오 → 평가를 고도화. 하/중/상을 딱 나누는 완벽한 기준은 없다.

## 2. 시스템 개요 · 업무 흐름

### 2.1 한눈에 보는 입력 → 출력

```
[입력] 신규 도서 서지정보 (BookInput)
   예) 제목「가장 인간적인 도시」 · 부제「…AI시대 건축」 · 저자 정현재 · 082=720.105
    │
    ▼  ① 1차 retrieve  (recall 넓게)   → 후보 청구기호 + 각 번호대 샘플 책
    │     예) 후보 {720.105(082), 720.2, 711.4} + 각 번호대 책 제목들
    ▼  ② 2차 classify (Claude, 판단)   → ▼h 후보 + 근거 + 에스컬레이션
    │
[출력] ClassificationResult  (구조화 JSON, 자유 텍스트 한 줄 아님)
   예) [1] 720.2 도시·건축(교양) 0.60  [2] 720.105 …  → escalate: needs_toc(HITL)
```

- **입력** = 서지정보 JSON 한 건. **출력** = ▼h 후보 목록 + 근거 + HITL 여부(구조화). 계약은 §3.
- 실행: `python src/run.py [골든셋경로]` → 콘솔 출력 + `output/`에 json 덤프.

### 2.2 실제 사서 업무 흐름 (이걸 이식한다)

사서는 082(또는 후보번호)를 **본교(로욜라)에서 확인**하고, **더 큰 풀(종합서지)에서 분포**를 본 뒤, **다시 본교로 되돌려** 적합을 확인한다:

```
082 확인
 ① 청구기호 검색 → 본교서지 : 082(또는 후보번호)가 본교에서 뭘 의미하나 / align 하나
 ② 제목 검색   → 종합서지 : 더 큰 풀(타학교)에서 유사 도서가 어느 번호에 몰리나 (분포)
 ③ 청구기호 검색 → 본교서지 : ②의 후보 번호를 다시 본교로 가져와 의미·적합 확인
 ④ 확정
```

### 2.3 우리 Ver1 근사 (②+③을 본교 하나로 합침)

로욜라는 **본교서지**라 종합서지가 없다. 그런데 사서 결론(부록2): *"종합서지는 후보 제안만 할 뿐 판단 권한 없음. 후보 제안은 본교 제목·키워드 검색으로도 됨 = 종합서지 꼭 필요 없음."* → **②의 종합서지를 본교 DB 제목검색으로 대체**해 ②+③을 하나로 합친다:

```
제목 검색 → 본교 DB → 유사 도서들의 청구기호 = 후보들
   → 각 후보의 '의미'는 이미 본교 기준 (내 DB니까)
   → LLM이 분포(개수) + 의미(주제 적합) + 규칙으로 최종 판단
```

> ⚠️ **본교-only의 한계:** 본교에 책이 있는 번호만 후보로 낼 수 있다(케이스5의 720.563은 본교 0권이라 후보에서 빠짐 — 대개 정합이나, 새 주제로 본교가 빈약하면 후보를 놓쳐 상 케이스/HITL로 감). 담당 사서 확인 권장.

### 2.4 funnel 구조 (recall → precision)

082 유무로 진입 경로가 갈리고(082 있음 → 청구기호 검색부터 / 없음 → 제목 검색부터), 에이전트는 이를 **말로 설명하며** 진행한다(XAI). 전체는 **깔때기**다:

```
1682건 (collection.json = 본교 장서 풀)
   │  ① 1차 retrieve (recall, 넓게·싸게)          RetrieveResult
   │     제목/부제/키워드로 후보 번호 건짐 (+082 prior). 각 번호대 샘플 책 첨부.
   ▼
후보 번호 ~N개 (+ 각 번호대 대표 책들)
   │  ② 2차 classify (precision, LLM)             ClassificationResult
   │     샘플 책으로 '주제·급'을 파악해 ▼h 판단. 개수는 보조. 애매하면 escalate.
   ▼
▼h 후보 1~3개 + 근거 (+ 필요시 HITL)
```

- **1차는 정확할 필요 없다(recall 우선).** threshold를 칼같이 정하지 않고 넉넉히 건진다. 정답 판단은 2차.
- **1차 = 두 구현, 같은 계약(RetrieveResult).** 내부만 다르고 배선/2차는 불변. `config.RETRIEVER`로 스위치, evaluate로 비교.
  - `retrieve_token` (V1): 제목 토큰 겹침 ≥ `MIN_OVERLAP`. 싸고 단순. 제목 안 겹치면 못 잡음.
  - `retrieve_embed` (V2): 임베딩 의미 유사("AI시대"↔"인공지능시대"). 백엔드 결정 후 구현.
- **LLM 호출은 classify 한 곳에 집중.** 제목 유사도는 후보를 건지는 그물일 뿐, 결정은 2차가 주제로.

> **주제 우선 · 개수 보조 (원칙).** 후보 번호의 '개수'는 결정자가 아니다: (a) DB는 번호당 최대 50건 **샘플**이라 개수 부정확(잘림 → `crawl_coverage.json`), (b) 개수 최다가 정답 아닌 케이스 존재(케이스11: 179.9 최다이나 오답). → 2차는 shelf_sample로 **주제 적합**을 최우선 판단.

## 3. 데이터 계약 (schema) — `src/schema.py`

단계 간에 오가는 타입. 내부 구현이 바뀌어도 이 계약은 고정한다. (**최우선 확정 대상**)

### 3.1 BookInput (입력)
```jsonc
{
  "title": "가장 인간적인 도시",
  "subtitle": "실리콘밸리의 젊은 건축가가 안내하는 AI시대 건축",
  "author": "정현재",
  "ddc_082": "720.105",     // 선택: 업체 DDC(prior). 없으면 null
  "is_translation": false,
  "original_title": null, "original_lang": null,
  "keywords": [], "toc": [], "description": null   // 상 케이스에서만 채워질 수 있음
}
```
> ⚠️ `keywords`의 출처는 미정(열린 질문 §7). 현재 골든셋 keywords는 셋업 시 수기로 넣은 것 → analyze 단계(LLM 키워드 추출)로 정식화 필요.

### 3.2 RetrieveResult (① 1차 산출물)
```jsonc
{
  "retriever": "token",              // token | embed
  "query_terms": ["도시","건축", ...],
  "messages": ["082(720.105) 있음 → …"],   // 진행 멘트(XAI)
  "total_title_hits": 4,
  "candidates": [                    // 후보 청구기호 + 근거
    { "ddc_h": "720.2", "is_082_prior": false,
      "title_hit_count": 3,          // 제목 매칭이 이 번호 가리킨 수
      "shelf_count": 50,             // DB 이 번호대 책 수(상한 샘플 → 부정확)
      "title_hits": ["…"], "shelf_sample": ["…"] }  // 주제 파악용 책 제목
  ]
}
```

### 3.3 ClassificationResult (② 2차 산출물 = 출력)
```jsonc
{
  "candidates": [                    // 적합도 내림차순
    { "h": "720.2", "label": "도시·건축(교양)", "confidence": 0.60,
      "source": "title_match",       // 082_kept | 082_rejected | title_match | inherited
      "reasoning": "…대중 교양서라 교양 서가…", "similar_refs": ["…"] }
  ],
  "ambiguity": 0.3,                  // 0(명확)~1(모호)
  "escalate": true,                  // HITL 필요?
  "signals": ["needs_toc"],          // cross_major | needs_toc | scattered_dist | org_disagree
  "notes": "종합 판단 근거 요약"
}
```

## 4. 구성 모듈 상세

> 경계 = "정해진 입력 → 정해진 출력"의 순수 함수. 내부만 갈아끼운다. (교수님 설계 사상 계승)

| 모듈 | 역할 | 입력 → 출력 | 구현 요점 |
|---|---|---|---|
| `crawler/callno_crawler.py` | DB 수집 | 청구기호 리스트 → collection.json/csv + coverage | 검색축 `st=FRNT`, `__verified_refresh` 쿠키 우회(Session), 번호당 5p 샘플, 잘림 로그 |
| `retrieve_token.py` (V1) | 1차 recall | BookInput → RetrieveResult | 제목+부제+키워드 토큰화, DB 제목과 겹침 ≥`MIN_OVERLAP`(컷 없이 넓게), 걸린 책의 ddc_h를 번호별 grouping, 082 prior 강제 포함 |
| `retrieve_embed.py` (V2) | 1차 recall | BookInput → RetrieveResult | 임베딩 의미 유사(백엔드 미정). 조어·동의어("AI시대↔인공지능시대") 잡기 목표 |
| `classify.py` ★ | 2차 판단 | BookInput + RetrieveResult → ClassificationResult | **유일한 LLM(Claude, `output_format` 강제).** shelf_sample로 주제+**급/장르** 판단, 개수 보조, 도메인 규칙 주입, 에스컬레이션 신호 판정 |
| `pipeline.py` | 배선 | BookInput → PipelineOutput | `config.RETRIEVER`로 1차 스위치 → 2차 연결 |
| `run.py` | 실행기(CLI) | 골든셋 경로 → 콘솔 + `output/*.json` | 케이스 실행, 1차/2차/합본 3종 덤프 |
| `evaluate.py` | 평가 하니스 | `scenarios/*.json` → Exact/Prefix/Top-k | 정답 숨기고 예측 후 대조. 검색기별 비교. gold 두 층(작성자/교열) |
| `schema.py` / `config.py` | 계약 / 설정 | — | §3 타입 정의 / 모델·경로·RETRIEVER·MIN_OVERLAP·SHELF_SAMPLE |

## 5. 난이도 = 고정 라벨이 아니라 "에스컬레이션 신호의 강도"

케이스 분석의 하/중/상은 **딱 나뉘는 정답 라벨이 아니다.** 부록1도 상(上)을 **트리거 신호**로 정의한다. 시스템은 난이도를 *입력으로 받지 않고* **검색 결과의 모양을 보고 스스로 판정**한다.

### 에스컬레이션 신호 (HITL 트리거)
1. 후보가 **성격 다른 대분류를 넘나듦** (문학 800 vs 부동산 330 vs 성찰 170)
2. **제목·부제·저자만으로 초점 판별 불가** → 목차/책소개 필요
3. **후보 분포가 흩어져 수렴 안 함**
4. **기관 간 분류 상이** (교보·알라딘)

| 등급 | 성격 | 에이전트 동작 |
|---|---|---|
| 하 | 규칙·서지 검색 명확히 수렴 | 자동 처리 |
| 중 | 082 기각/인접 선택 필요하나 서지 의미로 방어 가능 | 자동 처리 + 근거, 저신뢰 시 플래그 |
| 상 | 실질적 다의성, 사서마다 갈림 | **HITL 필수** — 단일 정답 강제 금지, 후보+근거형 |

> ▼h는 두 층위 정답을 가짐: `작성자 확정` vs `교열/최종`. 상 케이스는 둘이 다르다(뒤집힘).
> → 상 케이스 평가는 "정답 하나 맞히기"가 아니라 **Top-k + 에스컬레이션 정확도**로 본다.

## 6. 도메인 규칙 (케이스 분석에서 추출 → classify에 주입)

- **본교 서지 최우선.** 기존 서지/판본 있으면 그 청구기호 **그대로 승계**(결정론적). 타학교가 달라도 무시.
- **번역서:** 원제 검색 → 기존 원서/번역서 청구기호 **우선**, ▼m에 `K`. 외국인 저자는 항상 전거 생성 → 전거 없음 = 책 없음.
- **082는 prior.** 주제와 align 안 되면 기각하고 서지 의미/분포로 재선택.
- **종합서지는 후보 제안만.** 판단 권한 없음(본교 제목·키워드 검색으로 대체 가능).

## 7. 설계 질문 — 결정 현황

케이스5를 굴리며 정한 것들. 근거는 `devlog.md`, 관찰은 `output/`.

- **책 정보를 어느 정도 받나?** → `BookInput`(제목·부제·저자·082·키워드·목차·책소개). 목차/책소개는 상 케이스만.
- **제목 "비슷"을 어떻게? (하드코딩 vs LLM vs 임베딩)** → **funnel로 우회**(1차 recall 넓게, 판단은 2차). 토큰(V1)/임베딩(V2) 두 버전 → 평가로 결정. ⚠️ 한국어 토큰은 조사가 붙어("건축가가"≠"건축") 약함 → V2 근거.
- **LLM이 주제를 무엇으로?** → 후보 번호대 **shelf_sample로 주제·급 파악**. ⚠️ shelf_sample이 편향되면 판단도 편향 → 대표성이 지렛대(케이스5, `output/case05_*_analysis.md`).
- **개수를 신뢰?** → 아니오. **주제 우선·개수 보조**(§2.4).
- **키워드 출처?** → 미정. 현재 수기 → analyze 단계 정식화 필요.

## 8. 평가 (evaluate)

- **정량:** 골든셋의 교열 최종 ▼h 대비
  - **Exact**(완전 일치) — 엄격, 하/중 위주
  - **Prefix**(앞 3자리/대분류 일치) — **1차 우선 지표** (720.2↔720.105도 "720"은 맞음)
  - **Top-k**(후보 안에 정답 포함) — 상 위주
  - baseline: 검색 분포 최빈 ▼h(다수결) 대비 LLM+규칙 기여
- **정성:** 상 케이스에서 복수 후보·근거 타당성, 에스컬레이션 판정이 맞았나.
- **비결정성:** 애매 케이스는 top pick이 실행마다 흔들림(케이스5) → **Exact 대신 Top-k+에스컬레이션**이 신뢰 지표. (다중 샘플/저온도 고려)

### ⚠️ 데이터 누출 & 홀드아웃 (미구현, 다음 과제)
정답 **라벨**(expected)은 코드로 깔끔히 숨겨진다 — 파이프라인은 `input`만 받고 `BookInput`엔 정답 필드가 없다. **그러나** DB를 같은 도서관(로욜라)에서 크롤해서, 시나리오 입력 도서가 이미 장서에 있으면 **정답째로 코퍼스에 들어온다**(간접 누출). 실제로 케이스3「향신료의 역사」는 DB에 641.338309로 등재돼 있어 retrieve가 그 책을 그대로 찾아 "판단" 아닌 "승계"로 맞힘 → **점수 부풀림**. (케이스5·11 입력은 DB에 없어 누출 없음 → 진짜 난이도가 드러남.)
→ 공정 평가엔 **홀드아웃**(평가 대상 책을 DB에서 제외) 필요.
