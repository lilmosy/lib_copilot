# lib_copilot 설계서 (Ver 1)

> 도서관 청구기호(852 ▼h) 부여를 돕는 **사서-AI 협업 에이전트**.
> 실제 사서의 업무 흐름을 그대로 코드 로직으로 옮기고, 사람이 판단해야 하는 지점(HITL)만 남긴다.
> 기반: 교수님의 [`sogang_renAIssance`](../../../sogang_renAIssance) 설계를 참조하되, **실제 업무 프로세스**에 맞춰 다시 설계함.

---

## 0. 문제 선언

현재 청구기호(특히 852의 ▼h)를 정하는 과정은 **repetitive하고 비일관적**이다.

- **repetitive** → 자동화한다.
- **비일관적(사서 암묵지)** → 일관된 로직으로 대체하고, 사람은 **판단이 갈리는 지점에만** 개입한다.

목표: 시간 단축 + 암묵지(tacit)를 분명한 규칙으로 만들기.

### 왜 ▼h만인가 (스코프)
852의 다른 하위요소는 규칙/근사로 처리 가능하고, 진짜 "판단"이 갈리는 곳은 ▼h(분류기호)다.
그래서 Ver 1의 지적 난이도는 **▼h 결정 한 곳에 집중**한다. (▼i 도서기호·▼m 부가기호·852 조립은 이후 과제)

## 1. 시뮬레이션 전제 (중요)

이 프로젝트는 실제 도서관 시스템에 붙지 않는다. **시나리오 기반 시뮬레이션**으로 개발·평가한다.

- **정답셋(ground truth):** 고은님이 근무 중 수기 기록한 케이스 분석
  ([`../../../청구기호(h번호) 자동생성 · 의사결정 보조를 위한 케이스 분석.docx`]) — 12개 케이스.
  → `data/scenarios/`에 케이스별 골든셋 JSON으로 정규화한다.
- **DB 가정:** 업체 082를 실시간으로 받아오는 파이프는 만들지 않는다. 대신
  **"업무 과정에서 마주치는 책들"**(= 입력 도서가 아니라, 청구기호 검색·제목 검색으로 *떠오르는* 책들)을
  모아둔 DB가 있다고 가정한다. 이 DB(`data/collection.json`)에서 정답에 수렴하는 값을 만들어낸다.
  - 사서 결론과 정합적임: 본교 서지(로욜라 DB) 정합성이 최우선이고, 종합 서지(타학교)는 후보 제안만 할 뿐 판단하지 않는다.
    후보 제안은 본교 서지 내 제목·키워드 검색으로도 된다 → **DB 하나로 단순화 가능.**
- **개발 방식:** 사례 하나를 end-to-end로 굴려보며 시행착오 → 평가를 계속 고도화한다.
  하/중/상을 딱 나누는 완벽한 기준은 없다. 지금은 **케이스 5 하나**로 판정선을 찾는다.

## 2. 실제 업무 프로세스 = 파이프라인

하/중 난이도 케이스의 공통 패턴은 **파이프라인화가 가능**하다 (케이스 분석 부록1):

```
082 확인
  → 본교서지 청구기호 검색 (의미 확인)
  → 종합서지 제목 검색 (분포 확인)
  → 필요 시 키워드 검색
  → 확정
```

에이전트는 이 흐름을 **말로 설명하며** 진행한다(그 자체가 XAI). 082 유무로 진입 경로가 갈린다:

| 082 입력 | 첫 멘트 | 진행 |
|---|---|---|
| **있음** | `082(있음) → 청구기호 검색을 진행합니다` | 청구기호 검색(의미 확인) → 제목 검색(분포) → … |
| **없음** | `082(없음) → 청구기호 검색 없이 책제목 검색으로 들어갑니다` | 제목 검색(분포) → 키워드 검색 → … |

> 082는 **선택 입력(prior)**이다. 있으면 강한 힌트로 쓰되, 주제와 align 안 되면 **기각**할 수 있다.
> (없으면 생략. 시나리오 set에는 082가 적혀 있으므로 평가 시엔 대개 주어진다.)

### 파이프라인 단계 (src/)

### funnel 구조 (2단계)

실제 업무는 "제목으로 후보를 넓게 본 뒤, 그 번호들을 본교 기준으로 좁혀 판단"한다. 이를 **깔때기**로 옮긴다:

```
1682건 (collection.json = 본교 장서 풀)
   │  ① 1차 retrieve (recall, 넓게·싸게)          RetrieveResult
   │     제목/부제/키워드로 후보 번호를 건짐 (+082 prior). 각 번호대 샘플 책 첨부.
   ▼
후보 번호 ~N개 (+ 각 번호대 대표 책들)
   │  ② 2차 classify (precision, LLM)             ClassificationResult
   │     샘플 책으로 '주제'를 파악해 ▼h 판단. 개수는 보조. 애매하면 escalate.
   ▼
▼h 후보 1~3개 + 근거 (+ 필요시 HITL)
```

- **1차는 정확할 필요 없다(recall 우선).** threshold를 칼같이 정하지 않고 넉넉히 건진다. 정답 판단은 2차 LLM.
- **1차 = 두 구현, 같은 계약(RetrieveResult).** 내부만 다르고 배선/2차는 불변:
  - `retrieve_token` (V1): 제목 토큰 겹침 ≥ `MIN_OVERLAP`. 싸고 단순. 제목 안 겹치면 못 잡는 한계.
  - `retrieve_embed` (V2): 임베딩 의미 유사. "AI시대"↔"인공지능시대"를 잡음. 백엔드(Voyage/로컬 등) 결정 후 구현.
  - config.RETRIEVER 로 스위치. evaluate로 골든셋에서 V1 vs V2 비교.
- **제목 유사도는 "후보를 건지는 그물"일 뿐 정답 결정자가 아니다.** 결정은 2차가 주제로.
- LLM 호출은 **classify 한 곳에 집중**한다.

> **주제 우선 · 개수 보조 (원칙).** 후보 번호의 '개수'는 두 이유로 결정자가 될 수 없다:
> (a) DB는 번호당 최대 50건 **샘플**이라 개수가 부정확(상한에서 잘림 → crawl_coverage.json 참조).
> (b) 개수 최다가 정답이 아닌 케이스 존재(케이스11 부린왕자: 179.9 최다이나 오답).
> → 2차 프롬프트는 shelf_sample로 **주제 적합**을 최우선 판단하게 하고, 개수는 보조로만 쓴다.

## 3. 난이도 = 고정 라벨이 아니라 "에스컬레이션 신호의 강도"

케이스 분석은 하/중/상 3등급을 매기지만, 이는 **딱 나뉘는 정답 라벨이 아니다.**
부록1도 상(上)을 라벨이 아니라 **트리거 신호**로 정의한다. 그래서 시스템은 난이도를 *입력으로 받지 않고*,
**검색 결과의 모양을 보고 스스로 판정**한다("이건 상 같으니 HITL 띄운다").

### 에스컬레이션 신호 (HITL로 올려야 하는 트리거)
1. 후보 분류가 **성격이 다른 대분류를 넘나듦** (예: 문학 800 vs 부동산 330 vs 성찰 170)
2. **제목·부제·저자 메타데이터만으로 초점 판별 불가** → 목차/책 소개 필요
3. **종합서지 분포가 여러 번호로 흩어짐** (수렴하지 않음)
4. **기관 간 분류 상이** (교보·알라딘 등)

신호가 강하면: 단일 청구기호를 강제하지 말고 **후보 + 근거(+ 가능하면 교열 이력)**를 함께 제시한다.

| 등급 | 성격 | 에이전트 동작 |
|---|---|---|
| 하 | 규칙·서지 검색이 명확히 수렴 | 자동 처리 (rule + catalog lookup) |
| 중 | 082 기각/인접 선택 필요하나 서지 의미로 방어 가능한 단일 정답 | 자동 처리 + 근거 제시, 저신뢰 시 검토 플래그 |
| 상 | 실질적 다의성, 메타만으론 부족, 사서마다 갈림 | **HITL 필수** — 단일 정답 강제 금지, 후보+근거형 |

> ▼h가 두 층위의 "정답"을 가짐에 유의: `작성자 확정` vs `교열/최종`. 상 케이스는 둘이 다르다(뒤집힘).
> → 평가는 상 케이스에서 "정답 하나 맞히기"가 아니라 **후보 안에 최종답이 있나(Top-k) + 근거 타당성**으로 본다.

## 4. 도메인 규칙 (케이스 분석에서 추출)

classify 단계에 컨텍스트로 주입할 규칙들. (근거: 케이스 분석 문서)

- **본교 서지 최우선.** 기존 서지/판본이 있으면 그 청구기호를 **그대로 승계**(결정론적). 타학교가 다른 번호를 써도 무시.
- **번역서:** 원제 검색 → 기존 원서/번역서 청구기호 **우선**. ▼m란에 `K` 표시.
  외국인 저자는 항상 전거를 생성하므로 **전거 없음 = 책 없음**. 번역서는 원제 검색 규칙을 항상 태운다.
- **082는 prior.** 주제와 align 안 되면 기각하고, 서지 의미/분포로 재선택.
- **종합서지는 후보 제안만.** 판단 권한 없음. (본교 서지 제목·키워드 검색으로 대체 가능)

## 5. 설계 질문 — 결정 현황

케이스5를 굴리며 정한 결정들. 근거는 `docs/devlog.md`, 실행 관찰은 `output/`.

- **책 정보를 어느 정도 받나?** → `BookInput`(제목·부제·저자·082·키워드·목차·책소개). 목차/책소개는 상 케이스에서만 채움.
- **제목을 어느 정도 "비슷"으로 보나? (하드코딩 vs LLM vs 임베딩)** → **funnel로 우회.** 1차는 recall 넓게(threshold를 절대 컷으로 안 정함), 정답 판단은 2차 LLM. 방법은 두 버전(토큰/임베딩)으로 만들어 **평가로 결정**.
  - ⚠️ 한국어 토큰 매칭은 조사가 붙어("건축가가"≠"건축") 약함 → 임베딩(V2) 필요성의 근거.
- **LLM이 주제를 무엇으로 판단하나?** → 후보 번호대의 **shelf_sample(그 번호 책 제목들)로 주제 파악**. 제목/장르/책소개 중 택일이 아니라, "번호대의 실제 책들"이 주제 신호.
  - ⚠️ 관찰(케이스5): shelf_sample이 편향되면 LLM 판단도 편향된다(720.105를 'AI건축'으로 읽어 082 유지 → gold 720.2와 갈림). → **shelf_sample 대표성**이 설계 지렛대. `output/case05_humane_city_token_analysis.md` 참조.
- **개수를 신뢰하나?** → **아니오. 주제 우선·개수 보조** (§2 인용). 상한 샘플이라 부정확 + 개수최다≠정답.
- **본교-only 근사 (종합서지 없이)?** → 사서 결론(종합서지는 후보 제안만, 본교 제목검색으로 대체 가능)과 정합. 단 본교에 빈약한 새 주제는 후보를 놓칠 수 있음 → 상 케이스로 HITL. (담당 사서 확인 권장)

## 6. 평가 (evaluate)

- **정량:** 골든셋의 `최종 확정 ▼h`에 대해
  - Exact match (완전 일치) — 하/중 위주
  - Prefix match (앞 3자리 / 대분류 일치)
  - Top-k hit (후보 안에 최종답 포함) — 상 위주
  - baseline: "검색 분포 최빈 ▼h"(RAG 다수결) 대비 LLM+규칙 기여
- **정성:** 상 케이스에서 복수 후보 + 근거의 타당성을 사람이 검토. 에스컬레이션 판정이 맞았나.
- **로그:** `runs/<타임스탬프>.jsonl`에 케이스별 예측/정답/신호를 남긴다.

## 8. 빌드 순서

- **M0 — 스켈레톤 관통** ← 현재. schema 확정 + 시드 DB + 케이스5 골든셋 + 더미 배선(멘트까지).
- **M1 — 검색(retrieve)** 실제화. 케이스5에서 720.2 분포/개수가 그럴듯하게 나오는지.
- **M2 — classify(Claude)** 연결. 후보+근거+에스컬레이션 판정.
- **M3 — evaluate** 골든셋 리포트 → 오분류 분석 → 프롬프트/검색 개선 반복.
- **M4 — 크롤링 연동** 시드 DB를 실제 로욜라 검색 결과로 확장.
```
